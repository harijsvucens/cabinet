# Upstream Update Survival Strategy

## The Problem

Cabinet ships fast. Upstream updates (`v0.5.3` → `v0.5.4` → ...) happen frequently. When you run `git pull` against this repo (which IS the upstream repo — `cabinetai/cabinet`), any local source-code changes get destroyed by merge conflicts or overwrites. This has happened repeatedly.

## What Cabinet Already Protects (Safe)

These are gitignored from the main repo — upstream can never touch them:

| Path | Content | Notes |
|------|---------|-------|
| `data/` | All KB pages, agent configs, rooms, skills, conversations, DB | Has its own `.git` for version history |
| `.env.local` | Runtime config (KB_PASSWORD, DOMAIN, CABINET_DATA_DIR, ports, GITHUB_TOKEN) | Never tracked |
| `.cabinet.env` | Auto-generated per-install auth salt | Never tracked |
| `.cabinet-install.json` | Per-machine dataDir config | Never tracked |
| `skills-lock.json` | Per-machine skill bundle lock | Never tracked |
| `.agents/skills/` | Installed skill bundles | Never tracked |
| `.claude/` | Claude Code config | Never tracked |
| `.audit-shots/` | UI audit screenshots | Never tracked |

## What Gets Destroyed

Any modification to **tracked source files** goes away on `git pull` / upstream merge:

| Path | At-Risk? | Mitigation |
|------|----------|------------|
| `src/` | Yes | Don't edit. Contribute upstream instead. |
| `server/` | Yes | Don't edit. Contribute upstream instead. |
| `electron/` | Yes | Don't edit. |
| `scripts/` | Yes | Don't edit, or keep diffs as patch files. |
| `cabinetai/`, `cli/` | Yes | Don't edit. |
| `docs/` | Yes | Don't edit core docs. Add new docs in `docs/` if needed. |
| `package.json` | Yes | Don't edit. Use `.env.local` for config. |
| `package-lock.json` | Yes | Regenerated by `npm install`. |
| `next.config.ts` | Yes | Don't edit unless absolutely necessary. |
| `tsconfig.json` | Yes | Don't edit. |
| `CLAUDE.md` | Yes | Append custom instructions at the end. |
| `PROGRESS.md` | Yes | Append-only. Conflicts are easy to resolve. |

## Strategy: Two Paths

### PATH A (Recommended): Switch to `source-managed` Install

Cabinet was designed for this. The app code and your data live in completely separate places.

**How it works:**
- `npx create-cabinet@latest` downloads a **prebuilt standalone bundle** from GitHub Releases
- App code installs to `~/.cabinet/app/vX.Y.Z/` — managed by the installer, not git
- Your `data/` directory stays wherever you point `CABINET_DATA_DIR`
- Updating is a one-liner: `npx cabinetai update`
- The updater preserves `data/`, `.env.local`, and install metadata automatically
- Upstream code changes NEVER touch your data

**Migration steps (current source-custom → source-managed):**

```
1. Stop the running cabinet processes (Ctrl+C on dev:all)
2. Back up your data directory:
   cp -r data/ ~/cabinet-data-backup-$(date +%Y%m%d)/
3. Run the managed install in a NEW directory (do NOT re-use this git repo):
   npx create-cabinet@latest ~/cabinet-managed
   cd ~/cabinet-managed
4. Point it at your existing data:
   echo 'CABINET_DATA_DIR=/home/likkmrl/cabinet/data' > .env.local
   # OR copy data/ into the new install:
   # cp -r /home/likkmrl/cabinet/data ./
5. Start it:
   npm run dev:all
6. Verify everything works at http://localhost:4000
7. Once verified, you can archive the old git-clone directory
```

**AI agent update workflow (source-managed):**

```
# Check current version
npx cabinetai doctor

# See what's available
npx cabinetai update --check

# Apply update
npx cabinetai update

# If that fails, reinstall latest (data/ is preserved):
npx create-cabinet@latest ~/cabinet-managed
# then copy your data/ back or point .env.local at it
```

**Rule: NEVER edit files inside `~/.cabinet/app/`. That's the managed app directory. All customization goes in `data/`.**

### PATH B (If you MUST modify source): Fork + Rebase Workflow

Use this only when you need source changes that can't live in `data/` (e.g., custom API routes, UI modifications, daemon behavior).

**Setup (one-time):**

```
1. Fork cabinetai/cabinet on GitHub → your-username/cabinet
2. Clone your fork instead of upstream:
   git clone https://github.com/your-username/cabinet.git ~/cabinet-fork
   cd ~/cabinet-fork
3. Add upstream as a remote:
   git remote add upstream https://github.com/cabinetai/cabinet.git
4. NEVER commit to main. Create a branch for your customizations:
   git checkout -b my-cabinet
   # Make your changes, commit them here
```

**Updating to latest upstream:**

```
# AI agent: follow this exact sequence. DO NOT skip steps or improvise.

1. Stash or commit all current work on my-cabinet branch
2. git checkout main
3. git fetch upstream
4. git reset --hard upstream/main       # main = clean upstream
5. git push origin main --force          # sync your fork's main
6. git checkout my-cabinet
7. git rebase main                       # replay your changes on top
8. Resolve any conflicts (if any)
9. Run: npm install && npm run dev:all   # verify it works
10. If broken: git rebase --abort, then debug
11. git push origin my-cabinet --force   # update remote
```

**Conflict resolution rules:**
- `data/` conflicts: NEVER take upstream. Always keep YOUR data.
- `package.json`/`package-lock.json` conflicts: Take upstream, then `npm install`.
- `CLAUDE.md`: Accept upstream, then re-append your custom instructions.
- `PROGRESS.md`: Accept both sides (append).
- Source file conflicts: Manually resolve. Prefer upstream for core logic, keep your changes only where needed.

### PATH C (Current: Survive as source-custom with git stash)

If you stay on the direct clone of `cabinetai/cabinet`, use this safe update pattern:

```
# AI agent update workflow for source-custom:

1. git diff HEAD --name-only
   # Review: are any tracked files modified?
   # If only PROGRESS.md and CLAUDE.md are changed → safe to pull.
   # If src/, server/, scripts/, etc. are changed → continue below.

2. git stash push -m "pre-update-$(date +%Y%m%d)" --include-untracked

3. git pull --ff-only origin main
   # If this fails (not fast-forward): STOP. Something is wrong. Do NOT force.

4. git stash pop
   # Resolve any conflicts. Rules:
   #   data/  → keep stash (your data)
   #   PROGRESS.md → accept both
   #   CLAUDE.md → accept incoming, re-append custom lines
   #   src/, server/, etc. → manually resolve, prefer upstream

5. npm install

6. npm run dev:all
   # Verify the app starts and works

7. If broken:
   git stash       # re-stash your changes
   npm install     # make sure deps are clean
   npm run dev:all # verify upstream works clean
   # Then re-apply stash piece by piece
```

## General Rules for AI Agents Editing This Project

### DO
- Put all content and agent config in `data/` (gitignored, safe forever)
- Add new files under `docs/` for custom documentation (new files don't conflict)
- Use `.env.local` for all configuration (gitignored)
- Append to `PROGRESS.md` after every change (required by project convention)
- Append custom rules to end of `CLAUDE.md` (under a `## Custom Instructions` heading)
- Back up `data/` before any risky operation

### DO NOT
- Edit files in `src/`, `server/`, `electron/`, `cabinetai/`, `cli/`
- Modify `package.json`, `package-lock.json`, `tsconfig.json`, `next.config.ts`
- Modify core docs files unless adding a new document
- Run `git push` without explicit user instruction
- Delete or overwrite anything in `data/`
- Run `npm install` that changes `package-lock.json` unless after an upstream pull

### When Updating
1. Always check `git status` first
2. Back up `data/` before pulling upstream
3. Prefer `git pull --ff-only` (fail safe if not fast-forward)
4. After pull: `npm install`, verify with `npm run dev:all`
5. Report to user: new version, what changed, any conflicts

### Emergency Recovery
```
# If data/ gets corrupted:
git -C data/ log --oneline -10    # find last good commit in data repo
git -C data/ checkout <hash> -- . # restore data/ to that commit

# If source is broken after pull:
git stash                         # stash local changes
git reset --hard HEAD~1           # undo the pull
npm install                       # restore dependencies
npm run dev:all                   # verify it works
```
